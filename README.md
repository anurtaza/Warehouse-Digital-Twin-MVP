# 🏭 TwinStock — Real-time intelligence for your warehouse - MVP

3D цифровой двойник склада на Python + React + Three.js: симуляция WMS, погрузчики с водителями, FIFO/FEFO, маршрутизация по коридорам, WebSocket-обновления и адаптивный интерфейс для ПК и телефона.

## Быстрый старт

### Вариант 1 — Docker (рекомендуется)

```bash
docker compose up --build
```

Открыть: http://localhost:3000

В Docker фронтенд обслуживается через Nginx. API и Socket.IO проксируются через тот же порт `3000`:
- UI: `http://localhost:3000`
- API через фронтенд: `http://localhost:3000/api/state`
- backend напрямую: `http://localhost:5001`

### Вариант 2 — Вручную

**Бэкенд:**
```bash
cd backend
pip install -r requirements.txt
python app.py
# Сервер: http://localhost:5000
```

**Фронтенд (в другом терминале):**
```bash
cd frontend
npm install
npm run dev
# UI: http://localhost:3000
```

При ручном запуске Vite проксирует `/api` и `/socket.io` на backend. Если backend работает не на `http://localhost:5000`, задайте:

```bash
VITE_API_URL=http://localhost:5001 npm run dev
```

## Функциональность

### 3D-сцена склада
- Интерактивная визуализация всей карты склада 10×6.
- Ячейки хранилища с динамическим состоянием: заполнена, свободна, горячая зона.
- Погрузчики с водителями, визуальным маршрутом и целевой точкой.
- Открытый план склада без лишних перегородок — только стеллажи и коридоры.
- Маршруты рисуются по коридорам, с заметными стартом, финишем и стрелкой направления.
- Подсказки по ячейкам и погрузчикам при наведении.
- На телефоне камера Three.js отодвигается дальше, чтобы склад не был слишком приближен.

### Маршрутизация погрузчиков
- Оператор назначает водителю погрузчика целевую ячейку склада.
- Централизованный планировщик строит time-aware A* маршрут и резервирует ячейки во времени.
- Учитываются vertex/edge-конфликты, чтобы два погрузчика не занимали одну ячейку и не ехали навстречу по одному ребру.
- Визуализация траектории погрузчика в виде яркого коридорного маршрута на 3D-сцене.
- Плавное движение, разгон, короткие паузы на точках погрузки/разгрузки и корректный поворот по курсу.
- GUI-редактор маршрутов: выберите погрузчик, включите `Route`, кликните по ячейке склада.
- Операторские команды: `Pause`, `Resume`, `Charge`, `Cancel`.
- Отдельная панель строит маршрут складовщика через `/api/worker/route`.

### Auth / RBAC
- JWT-аутентификация для операторских действий.
- Роли:
  - `manager` — сценарии, оптимизация SKU, управление погрузчиками и назначение маршрутов.
  - `operator` — управление погрузчиками, водителями и назначение маршрутов.
  - `logistics` — WMS, заказы, SKU, throughput и отчёты без управления погрузчиками.
  - `viewer` — только просмотр дашборда.
  - `forklift` — водитель конкретного погрузчика, видит свою кабину, батарею, задачу и маршрут.
- Demo users:
  - `manager` / `managerpass`
  - `operator` / `operatorpass`
  - `logistics` / `logisticspass`
  - `viewer` / `viewerpass`
  - `forklift-0` / `forklift0pass`
  - `forklift-1` / `forklift1pass`
  - `forklift-2` / `forklift2pass`
  - `forklift-3` / `forklift3pass`

### Адаптивность
- Desktop-версия использует боковую панель и полноэкранную 3D-сцену.
- На экранах до `700px` панель превращается в выезжающее меню.
- На телефоне добавлена кнопка меню, затемнение фона и безопасные отступы для iPhone (`viewport-fit=cover`).
- Формы, карточки погрузчиков и кнопки перестраиваются под 360–390px.
- Tooltip на мобильном экране фиксируется снизу и не вылезает за ширину экрана.

### Панели и аналитика
- Режимы пользователя: менеджер склада, оператор, логист.
- Панель метрик в реальном времени: заполненность, активные погрузчики, заказы, скорость обработки.
- Сценарии: `normal`, `surge`, `agv_fail`, `low_staff`.
- Анализ сценариев и прогнозы производительности.
- WMS-панель с топ-SKU, инвентарем и очередью заказов.
- Экспорт отчёта в JSON.

### Симуляция склада и WMS
- Движок имитирует заполнение и опорожнение ячеек.
- Генерация заказов, смена статусов, синхронизация инвентаря.
- Логика погрузчиков с водителями, ожиданием задания, плавным движением и стабильным маршрутом.
- Ручной режим оптимизации размещения SKU по приоритету и расстоянию.
- Система событий с информационными и предупреждающими сообщениями.

### CCI / напитки
- FEFO/FIFO compliance: каждая ячейка хранит `received_at` и `expiry_date`, а WMS выбирает паллету с ближайшим сроком годности.
- Температурные зоны склада: `cold`, `ambient`, `dry` для разных категорий напитков.
- Тепловая карта активности: ячейки накапливают `activity_count` от пополнений, отгрузок и проходов погрузчиков.
- Предиктивное ТО погрузчиков: `wear_level`, `maintenance_due_at`, `maintenance_risk`.
- KPI-бенчмарки: fill target, throughput target, FIFO compliance, expiry risk, congestion и energy per shift.
- Прогноз запасов: WMS показывает, на сколько часов хватит топовых SKU при текущем темпе заказов.
- Сценарий `surge` адаптирован как “Летний сезон” с +400% заказов.

## API

| Метод | Эндпоинт | Описание |
|-------|----------|----------|
| GET | `/api/state` | Полное состояние склада |
| GET | `/api/metrics` | Только метрики |
| GET | `/api/events` | Лог событий |
| POST | `/api/login` | Получить JWT-токен |
| GET | `/api/me` | Проверить текущего пользователя |
| POST | `/api/scenario/{name}` | Переключить сценарий, manager only |
| POST | `/api/optimize` | Оптимизировать размещение SKU, manager only |
| POST | `/api/agv/{id}/command` | Pause/resume/charge/cancel для погрузчика, operator/manager |
| POST | `/api/planner/assign` | Назначить погрузчику маршрут к ячейке, operator/manager |
| POST | `/api/worker/route` | Построить маршрут складовщика, operator/manager/logistics |

**Сценарии:** `normal`, `surge`, `agv_fail`, `low_staff`

**WebSocket события:**
- `state` — полное обновление состояния
- `event` — новое событие в логе

class WarehouseSimulator:

        # Вместо симуляции — запрос к WMS API:
        return wms_client.fetch_current_state()

    def tick(self):
        # Polling реальных данных
        pass

Поддерживаемые протоколы подключения:
- REST API WMS (SAP EWM, 1C WMS, Manhattan)
- MQTT/OPC-UA от сенсоров и RFID

## Архитектура и реализация

### Основные функции и модули
- `backend/app.py` — Flask + SocketIO сервер, REST API, аутентификация по JWT, ролевая модель и симуляционная петля.
- `backend/simulator.py` — WMS-симулятор, генерация заказов, FIFO/FEFO логика, приоритетные срочные заявки, управление AGV, сценарии, метрики и события.
- `backend/planner.py` — централизованный маршрутизатор A* с временными резервациями для vertex/edge конфликтов.
- `frontend/src/App.jsx` — главный UI-компонент, 3D-сцена склада, визуализация AGV, маршруты, синхронизация состояния и обработка пользовательских команд.
- `frontend/src/components/*` — панели метрик, сценариев, WMS, событий, ролей, отчётов и аналитики.

### Что реализовано
- `FIFO/FEFO` для напитков: выбор ячеек по срокам годности и дате поступления.
- `expiry_date` в `Cell`, `zone_type` для температурных зон `cold/ambient/dry`.
- `urgent` заказы в пиковом сценарии `surge` с приоритетом назначения.
- `auto_dispatch()` в симуляторе: назначение свободных AGV на очередь WMS.
- Агент `AGV` с командной очередью, плавным движением, ускорением, блокировками и разгрузкой.
- Планировщик маршрутов с временными резервациями: `RoutePlanner.assign()` / `plan()`.
- Отдельное планирование маршрута складовщика через `/api/worker/route`.
- Метрики и инсайты: `fill_pct`, `throughput`, `fifo_queue`, `fifo_assigned`, `energy_kwh_shift`, `maintenance_risk`, `congestion_points`, `expiring_soon`.
- Сценарии `normal`, `surge`, `agv_fail`, `low_staff` для what-if анализа на уровне склада.
- Роли: `manager`, `operator`, `logistics`, `viewer`.
- WebSocket `state`/`event` — реальное время для UI.

### Задачи решаемые проектом
- Видимость склада и инвентаря в 3D.
- FIFO/FEFO compliance для напитков.
- Приоритетное назначение срочных заказов в пике.
- Предиктивное ТО по `wear_level` и `maintenance_due_at`.
- Мониторинг перегрузок зон и планировка горячих ячеек.
- Управление погрузчиками и маршрутизацией без конфликтов.
- Отчётность по KPI и прогноз запасов.
- Подготовка к интеграции с реальным WMS.

## Production deployment

### Запуск через Docker Compose

```bash
docker compose build --no-cache
docker compose up -d
```

- `backend` доступен на `http://localhost:5001`
- `frontend` доступен на `http://localhost:3000`

### Переменные окружения
- `JWT_SECRET` — заменить на секретную строку для токенов.
- `PORT` — если нужно изменить порт бэкенда.
- `FLASK_ENV=production` — используется в Docker Compose.
- `VITE_API_URL` — URL backend для отдельного hosted-фронтенда. В Docker Compose обычно не нужен, потому что Nginx проксирует `/api` и `/socket.io`.

### Как перевести проект в реальное производство

1. **Соберите контейнеры** с `docker compose build --no-cache`.
2. **Настройте секреты** (`JWT_SECRET`) через систему управления секретами.
3. **Добавьте обратный прокси** и HTTPS:
   - Nginx/Traefik/Load Balancer на порту 80/443.
   - Проксирование фронтенда и бэкенда.
4. **Разделение статики и API**:
   - фронтенд собирается `npm run build` и обслуживается Nginx;
   - бэкенд работает на Flask/SocketIO в контейнере.
5. **Стабильность**:
   - `restart: unless-stopped` в docker-compose;
   - мониторинг контейнеров и логов.
6. **Производственная интеграция WMS**:
   - замените или адаптируйте `backend/simulator.py` на реальный коннектор;
   - реализуйте методы `get_state()`, `tick()`, `wms.to_dict()` и `complete_order()` через API/БД.
7. **Дополнительно**:
   - используйте `docker compose up -d` в `production` окружении;
   - разворачивайте на Kubernetes / ECS / Docker Swarm при необходимости.

### Рекомендации для CCI / Coca-Cola
- Храните `expiry_date` и `zone_type` в реальном WMS/ERP.
- Оставьте FIFO/FEFO как обязательный порядок отгрузки.
- Используйте пиковый сценарий `surge` для стресс-тестирования загрузки склада.
- Подключите метрики нагрузки и конгестии для оперативных диспетчеров.
